技术栈：Android + Kotlin

## 一、项目概述

### 1.1 项目功能

本项目是一个多频道单列信息流展示应用，核心功能包括：

| 功能模块 | 功能描述 |
|---------|---------|
| 用户登录注册 | 支持用户名密码登录和注册 |
| 三频道信息流 | 精选、推荐、本地三个频道，通过 ViewPager2 切换 |
| 内容卡片 | 大图、小图、视频三种卡片类型，支持点赞、收藏 |
| 标签过滤 | 点击卡片标签进行过滤，跨频道同步（观察者模式） |
| 详情页 | 展示完整内容，视频播放控制，分享功能 |
| 视频播放 | 全局单例管理，支持 AndroidVideoCache 缓存播放 |
| AI 智能搜索 | DeepSeek API 对话，理解意图后检索数据库 |
| 用户中心 | 浏览历史、点赞列表、收藏列表的展示与跳转 |
| 页面动画 | Activity 切换动画、搜索提示滚动动画等 |

## 五、标签过滤模块（FilterManager）

文件：`FilterManager.kt`

**设计模式**：观察者模式

通过 `selectedTags` 集合存储选中的标签，订阅者通过 `addListener()` 注册监听。标签变化时通过 `notifyListeners()` 通知所有订阅者。

**主要订阅者**：
- **MainActivity**：更新顶部选中标签的显示
- **BaseChannelFragment**（三个频道实例）：重新加载并过滤数据

**触发方式**：点击 FeedAdapter 中卡片的标签 → `FilterManager.toggleTag(label)`

**过滤逻辑**（在 BaseChannelFragment 的 loadNextPage 中）：
- 有过滤标签时，加载全部数据 → `filter { it.label.any { label -> tags.contains(label) } }`
- 无过滤标签时，正常分页加载 `dao.getItemsSync(channelType, page * pageSize, pageSize)`

## 六、视频播放管理模块

文件：`util/VideoPlaybackManager.kt`

**设计模式**：全局单例

**核心设计原则**：任何时刻只允许一个视频处于播放状态，避免多个视频声音混杂和资源冲突。

### 6.1 核心方法

| 方法 | 功能 |
|-----|-----|
| `getProxyUrl(context, originalUrl)` | 获取带缓存的代理 URL |
| `isCached(context, url)` | 判断视频是否已缓存 |
| `play(itemId, videoView, playButton, coverView, videoUrl)` | 播放视频（自动暂停之前的视频） |
| `pauseCurrent()` | 暂停当前视频，显示播放按钮和封面 |
| `resumeCurrent()` | 恢复播放，隐藏封面 |
| `stop()` | 停止播放并清理状态 |
| `getCurrentPlayingId()` | 获取当前播放视频 ID |

### 6.2 视频缓存机制

使用 **AndroidVideoCache**（`com.danikula:videocache:2.7.1`），构建本地 HTTP 代理服务：

- **缓存目录**：`context.cacheDir`
- **最大缓存大小**：512 MB
- **最大缓存文件数**：20 个
- **代理 URL**：`http://127.0.0.1:<port>/<originalUrl>`

**工作流程**：
1. 首次播放：边下载边播放 → 写入本地缓存
2. 再次播放：直接从本地缓存读取 → 零网络消耗

### 6.3 BaseChannelFragment 中的视频智能暂停

RecyclerView 滚动时，检测当前播放视频是否仍在可见区域，若超出可见范围则自动调用 `VideoPlaybackManager.pauseCurrent()`。

## 七、AI 智能对话模块

### 7.1 架构概览

三层架构：UI 层（DialogActivity）→ 网络层（AiClient + AiService）→ 数据层（SearchService）

**API 服务**：DeepSeek API（`https://api.deepseek.com/v1/`）

**核心依赖**：
- `com.squareup.retrofit2:retrofit:2.9.0`
- `com.squareup.retrofit2:converter-gson:2.9.0`
- `com.squareup.okhttp3:okhttp:4.12.0`
- `com.squareup.okhttp3:logging-interceptor:4.12.0`

### 7.2 完整对话流程

1. **用户输入**：点击搜索框进入 DialogActivity，输入消息
2. **构建 Prompt**：AiClient.buildPrompt() 构建系统提示词（包含可用检索方法、标签说明、输出 JSON 格式）+ 用户消息
3. **API 请求**：DeepSeek API → model = "deepseek-chat"，Authorization: Bearer <API_KEY>
4. **响应解析**：尝试直接解析 JSON → 失败则从文本中提取 `{...}` → 仍失败则将全文作为纯文本回复
5. **数据库检索**：SearchService.executeSearch() 根据 AI 返回的 searchMethod 调用对应检索方法
6. **UI 展示**：显示 AI 纯文本消息（AI）或 AI 文本 + 搜索结果卡片（AI_WITH_SEARCH）
7. **点击卡片**：跳转到 DetailActivity 查看完整内容

### 7.3 Prompt 模板

```
系统指令：
- 你是智能助手，帮助用户通过标签检索广告内容
- 可用检索方法：searchByTag / searchByMultipleTags / searchByChannel / searchByKeyword
- 数据库标签：运动、旅行、科技、美食、美妆、家居 等
- 输出格式：{"reply": "...", "search_method": "...", "tags": ["..."]}
- 不需要检索时 search_method = "none"，tags = []

用户消息：<user_message>
```

### 7.4 SearchService 检索方法

文件：`dialog/SearchService.kt`

| 检索方法 | 功能 | 逻辑 |
|---------|-----|-----|
| `searchByTag(tag)` | 单标签匹配 | 全部频道数据 → 任一标签包含 tag |
| `searchByMultipleTags(tags)` | 多标签匹配 | 任一标签命中即返回 |
| `searchByChannel(channel)` | 按频道检索 | 直接查询该频道，最多 10 条 |
| `searchByKeyword(keyword)` | 关键词匹配 | 标题/描述/详情内容模糊匹配 |

## 九、卡片动态化方案

### 9.1 方案设计背景

**核心难点**：一个信息流列表中需要展示三种完全不同布局的卡片（大图卡片占满整行、小图卡片左右分栏、视频卡片带播放控制），同时还需要支持底部"已加载全部"提示视图。三种卡片的交互差异大：视频卡片需要处理播放/暂停状态切换，大图/小图卡片需要处理点赞/收藏动画。如何在一个 RecyclerView 中高效、可扩展地支持多种异构视图类型，是本模块需要解决的核心问题。

### 9.2 备选方案对比

#### 方案 A：多 ViewType + 独立 ViewHolder（最终采用方案）

**思路**：在 Adapter 中通过 `getItemViewType(position)` 返回不同类型值，`onCreateViewHolder` 根据 viewType inflate 不同的布局文件创建不同的 ViewHolder 子类，`onBindViewHolder` 根据 holder 类型调用不同的绑定逻辑。

**实现要点**（见 `FeedAdapter.kt:51-81, 94-127`）：

**优点**：
- **性能最优**：RecyclerView 根据 viewType 做 View 回收，相同类型的 ViewHolder 互相复用，避免了不必要的布局 inflate
- **类型安全**：每个 ViewHolder 有明确的字段绑定（`findViewById` 只在创建时执行一次，绑定结果保存在 ViewHolder 属性中），避免了 onBind 时反复查找
- **解耦清晰**：三种卡片的绑定逻辑各自独立在内部类中，修改某一种卡片不会影响其他类型
- **易扩展**：新增卡片类型只需添加一个常量 + 一个 ViewHolder 内部类 + 一个布局文件，无需改动现有代码

**缺点**：
- **代码量较大**：每种类型都需要独立的 ViewHolder 类和布局文件，4 种类型就有 4 个内部类
- **有一定样板代码**：点赞/收藏按钮的逻辑在 Big/Small/Video 三种 ViewHolder 中存在重复（当前实现中确实如此——三处都有相同的点赞按钮 onClick 逻辑）
- **视频状态管理略复杂**：视频卡片的播放状态需要和全局 VideoPlaybackManager 同步，bind 时需要判断当前 itemId 是否就是正在播放的视频

#### 方案 B：单一通用布局 + 条件可见性控制

**思路**：所有卡片共用一个布局文件，通过 `visibility = VISIBLE/GONE` 控制大图/小图/视频区域的显示。例如布局中同时包含 bigImageContainer、smallImageContainer、videoContainer 三个容器，onBind 时根据 type 字段设置某个容器 VISIBLE，其余 GONE。

**优点**：
- 代码量最少，只需 1 个 ViewHolder 类 + 1 个布局文件
- 新增类型时只需在布局中新增容器并在 onBind 中加一个分支

**缺点**：

- **measure/layout 成本高**：RecyclerView 在滚动时需要频繁 measure，深层嵌套的不可见 View 也会参与测量（虽然最终尺寸为 0，但测量过程仍有开销）
- **动画效果差**：GONE/VISIBLE 切换没有平滑过渡，用户在滚动中可能看到闪烁
- **可维护性差**：一个布局文件几百行，三种卡片的 UI 混在一起，修改一个字段容易影响其他类型


### 9.3 最终方案：多 ViewType + 独立 ViewHolder

**Footer 与数据列表的边界处理**（`FeedAdapter.kt:57-81`）：Footer 是一个"虚拟" item，不对应 AdItem 数据。通过 `showFooter` 标志位控制 `itemCount = dataCount + if(showFooter) 1 else 0`，`getItemViewType` 在最后一个位置时返回 TYPE_FOOTER。这保证了 footer 不会占用 DiffUtil 的计算范围。

## 十、列表性能策略

### 10.1 方案设计背景

**核心难点**：信息流列表的性能瓶颈体现在三个方面：
1. **滚动流畅度**：卡片包含图片加载（网络 IO + 解码）、视频播放（MediaPlayer 初始化/释放）、动态文本渲染，滚动时需要保证 60FPS
2. **数据一致性**：子线程（IO）加载数据、主线程渲染 UI，两个线程对数据列表的修改需要严格同步，否则 RecyclerView 会报 "Inconsistency detected" 崩溃
3. **状态更新效率**：点赞/收藏是高频率操作，如果每次都刷新整个列表（`notifyDataSetChanged()`），会导致所有可见 View 重新绑定 + 图片重新加载，体验极差

### 10.2 备选方案对比

#### 方案 A：ListAdapter + DiffUtil（最终采用方案）

**思路**：继承 `ListAdapter<AdItem, ViewHolder>(diffCallback)`，每次数据更新时调用 `submitList(newList)`。ListAdapter 内部会在后台线程执行 DiffUtil.calculateDiff，计算出旧列表和新列表的差异（新增/删除/修改项），然后自动调用 `notifyItemInserted/Removed/Changed` 做局部刷新。

**DiffCallback 定义**（`FeedAdapter.kt:41-49`）：

```kotlin
private val diffCallback = object : DiffUtil.ItemCallback<AdItem>() {
    override fun areItemsTheSame(oldItem: AdItem, newItem: AdItem): Boolean {
        return oldItem.id == newItem.id         // 身份判断：主键相同视为同一条
    }
    override fun areContentsTheSame(oldItem: AdItem, newItem: AdItem): Boolean {
        return oldItem == newItem                // 内容判断：data class 自动生成的 equals
    }
}
```

**优点**：
- **自动差异计算**：无需手动调用 notifyXxx，避免漏写/错写
- **后台线程计算**：DiffUtil.calculateDiff 在后台线程执行，不会阻塞 UI
- **局部刷新**：只有变化的 item 才会重新 onBind，图片不会被无意义重新加载（Glide 内部也会根据 URL 命中内存缓存，不会重新下载）
- **默认动画**：RecyclerView 自带 ItemAnimator，局部更新会有淡入淡出/平移动画，视觉体验更好
- **分页追加友好**：`submitList(previousList + newPageItems)` 可以高效地追加新页面数据，不需要刷新已有数据

**缺点**：
- **数据必须不可变**：每次 submitList 必须传入一个**新的列表对象**，不能修改已有列表再 submit（因为 DiffUtil 需要对比旧列表和新列表）。这要求开发者养成"创建新列表替换"的习惯，而不是"修改原列表"
- **Footer 特殊处理**：Footer 不是数据项，ListAdapter 本身不支持非数据 item，需要手动覆盖 `itemCount` 和 `getItemViewType`，并自己维护 `showFooter` 标志

#### 方案 B：BaseAdapter + 手动 notifyDataSetChanged

**思路**：直接使用 `RecyclerView.Adapter`，维护一个 `var items = mutableListOf<AdItem>()`，每次数据变化后调用 `adapter.notifyDataSetChanged()` 让整个列表重绘。

**优点**：
- 实现最简单，只需要一个可变列表 + 一行 notify 代码
- 无需考虑 DiffUtil 的不可变列表约束

**缺点**：
- **性能极差**：`notifyDataSetChanged()` 会让所有可见 item 重新执行 onBindViewHolder → 重新加载图片 → 重新测量布局。滚动中的"下拉加载更多"如果用此方案，新数据到达的瞬间会出现明显的卡顿
- **动画丢失**：没有局部动画，用户看不到"哪些数据是新加的"
- **用户体验灾难**：用户正在滚动的列表被强制刷新，滚动位置可能被打乱；图片加载中的卡片会被重置


### 10.3 最终方案：ListAdapter + DiffUtil + 手动 Footer 管理

**选择理由**：
1. **在复杂度和性能之间取得平衡**：比 BaseAdapter 性能好得多，比 Paging 3 简单得多
2. **DiffUtil 是 Android 列表优化的"基本功"**：学习项目的目标之一就是掌握这项技术
3. **Footer 问题可以通过少量代码解决**：通过 `submitListAndFooter` 自定义方法同时提交数据和 footer 状态，保证一致性

### 10.4 关键实现细节

**数据列表和 Footer 状态原子提交**（`FeedAdapter.kt:66-81`）：这是项目踩坑后得出的最重要经验。原先代码是分别调用 `submitList(list)` 和 `notifyDataSetChanged()` 来更新 footer，存在时序问题。改造后使用单一入口：

```kotlin
fun submitListAndFooter(list: List<AdItem>, showFooterValue: Boolean) {
    if (showFooter == showFooterValue) {
        submitList(list)              // footer 没变化，走标准 diff 流程
        return
    }
    showFooter = showFooterValue
    submitList(list)
    notifyDataSetChanged()            // footer 变化改变 itemCount，需要整体刷新
}
```

**Fragment 中所有数据更新切到主线程原子执行**（`BaseChannelFragment.kt:281-291`）：IO 线程查询数据库拿到数据后，**不**在 IO 线程修改 dataList，而是切到主线程再一次性完成"清空旧数据 → 追加新数据 → 提交 adapter"三步：

```kotlin
lifecycleScope.launch(Dispatchers.Main) {
    hasMore = moreAvailable
    dataList.addAll(newItems)           // 所有对 dataList 的修改都在主线程
    val finalList = dataList.toList()   // 转不可变列表传给 adapter
    val showFooter = !hasMore && finalList.isNotEmpty()
    adapter.submitListAndFooter(finalList, showFooter)
}
```

这样做保证了：**RecyclerView 在 layout/measure 过程中，dataList 的状态不会被另一个线程改变**，从根源上避免了 `Inconsistency detected` 崩溃。

**Glide 图片加载的占位与错误处理**（`FeedAdapter.kt:143-148`）：
- `placeholder(R.drawable.ic_launcher_background)`：图片加载中显示占位图，避免空白
- `error(R.drawable.ic_launcher_background)`：加载失败（网络错误/URL 无效）时显示错误图
- `centerCrop()`：确保图片按比例填满 ImageView，避免卡片高度跳动

**视频卡片滚动时自动暂停**（`BaseChannelFragment.kt:227-250`）：在 RecyclerView 的 `onScrolled` 回调中检查当前播放视频的 item 位置是否仍在屏幕可见范围内，如果超出则调用 `VideoPlaybackManager.pauseCurrent()`。这避免了用户滚动走后视频仍在后台消耗流量和 CPU。

**点赞动画不被打断的技巧**（`FeedAdapter.kt:477-480`）：先更新数据库（异步，不阻塞 UI），再启动动画。动画绑定在具体的 ImageView 上，只要这个 ViewHolder 没有被回收（DiffUtil 局部刷新保证了未变化的 item 不会被重新绑定），动画就能完整播放。

## 十一、AI 输出约束与缓存策略

### 11.1 方案设计背景

**核心难点**：
1. **LLM 输出不稳定**：DeepSeek 是通用大语言模型，即使在 Prompt 中明确要求"输出 JSON 格式"，模型仍可能输出自然语言解释、Markdown 代码块包裹的 JSON、甚至完全不符合格式的文本。客户端需要一套"防御性解析"机制。
2. **API Key 安全**：API Key 硬编码在源码中会被反编译提取，需要从外部注入。
3. **检索路由不可控**：模型可能返回不存在的 searchMethod（如"search_by_tag"而不是约定的"searchByTag"），或返回 tags 为空但 search_method 不为"none"，需要容错处理。
4. **数据库检索性能**：SearchService 每次查询都加载全部数据到内存做 filter，虽然当前数据量小可以接受，但设计上需要考虑扩展性。

### 11.2 备选方案对比

#### 方案 A：Prompt 强约束 + 三层解析降级（最终采用方案）

**思路**：
- **Prompt 层**：系统提示词中明确列出可用方法名、标签枚举、JSON schema，缩小模型的输出空间
- **解析层**：三层降级——① 直接 Gson 解析 → ② 从文本中正则提取 `{...}` → ③ 将全文作为纯文本回复（searchMethod=none）
- **路由层**：SearchService.executeSearch 对 searchMethod 做 lowercase + 归一化匹配，命中不到则返回空列表

**Prompt 约束设计**（`AiClient.kt:49-75`）：

```
你是智能助手，帮助用户通过标签检索广告内容
可用检索方法：searchByTag / searchByMultipleTags / searchByChannel / searchByKeyword
数据库标签：运动、旅行、科技、美食、美妆、家居 等
输出格式：{"reply": "...", "search_method": "...", "tags": ["..."]}
不需要检索时 search_method = "none"，tags = []
```

**三层解析实现**（`AiClient.kt:100-139`）：

```kotlin
// 第一层：尝试直接把模型输出当 JSON 解析
try {
    val aiResponse = gson.fromJson(aiReply, AiResponse::class.java)
    callback(Result.success(aiResponse))
} catch (e: Exception) {
    // 第二层：从文本中提取第一个 { 到最后一个 } 之间的内容
    try {
        val extractedJson = extractJsonFromText(aiReply)
        if (extractedJson != null) {
            val aiResponse = gson.fromJson(extractedJson, AiResponse::class.java)
            callback(Result.success(aiResponse))
        } else {
            // 第三层：无法解析，把全文当回复，不做数据库检索
            callback(Result.success(AiResponse(aiReply, "none", emptyList())))
        }
    } catch (e2: Exception) {
        callback(Result.success(AiResponse(aiReply, "none", emptyList())))
    }
}
```

**优点**：
- **健壮性极强**：无论模型输出什么，最终都能给出一条用户可见的消息，不会让 DialogActivity 崩溃
- **防御式编程**：不假设模型一定遵守格式，而是在每一步都考虑"失败了怎么办"
- **实现简单**：只需要几行 try-catch 和一个正则提取函数，不需要引入额外依赖

**缺点**：
- **依赖模型的"听话程度"**：如果模型完全拒绝输出 JSON（比如回复"我不能帮助你检索广告"），用户只能看到一段不相关的自然语言回复
- **没有结构化校验**：即使 JSON 解析成功，也不保证 `searchMethod` 是合法值、`tags` 列表中的标签确实存在于数据库中——这些校验在 SearchService 做了

#### 方案 B：Function Calling / 工具调用模式

**思路**：利用大模型的"工具调用"能力（如 DeepSeek 的 `/chat/completions` 支持 `tools` 参数），让模型输出一个结构化的 tool_call 对象而不是自由文本。这样客户端可以保证拿到的是严格符合 JSON Schema 的数据结构。

```kotlin
// 伪代码：用工具调用方式
val request = mapOf(
    "model" to "deepseek-chat",
    "messages" to [...],
    "tools" to [
        "type" to "function",
        "function" to mapOf(
            "name" to "searchByTag",
            "description" to "通过单个标签检索广告",
            "parameters" to mapOf("type" to "object", "properties" to mapOf(
                "tag" to mapOf("type" to "string", "description" to "标签名，如运动、科技")
            ), "required" to ["tag"])
        )
    ]
)
```

**优点**：
- **输出 100% 结构化**：模型输出的是严格符合 JSON Schema 的函数调用参数，不需要解析降级
- **可以定义多个工具**：searchByTag/searchByChannel 等是独立的工具，模型可以精确选择
- **是 LLM 应用开发的主流方向**：生产级 AI Agent 应用普遍使用工具调用

**缺点**：
- **DeepSeek API 的工具调用能力需要验证**：不是所有模型/版本都稳定支持 tools 参数
- **实现复杂度高**：需要额外处理 `finish_reason = "tool_calls"` 的响应，再手动调用工具，最后把工具结果作为一条消息回传给模型继续生成自然语言回复——这意味着一次用户输入需要发起两次 API 请求
- **Prompt 工程更复杂**：需要为每个工具写描述（description 写得好不好直接影响模型判断用哪个工具）

**为什么不选**：小型学习项目追求快速验证，方案 A 一次请求就能完成。方案 B 适合生产级应用，在需要高可靠性的场景下是更好的选择。

#### 方案 C：客户端意图识别（不依赖 LLM）

**思路**：完全不用 LLM，在本地实现一个简单的关键词匹配规则引擎：

```kotlin
fun routeQuery(userInput: String): AiResponse {
    return when {
        "运动" in userInput -> AiResponse("为你找到运动相关广告", "searchByTag", listOf("运动"))
        "推荐" in userInput -> AiResponse("为你精选推荐频道内容", "searchByChannel", listOf("1"))
        userInput.length < 3 -> AiResponse("请输入更多关键词", "none", emptyList())
        else -> AiResponse("为你搜索关键词", "searchByKeyword", listOf(userInput))
    }
}
```

**优点**：
- **零网络依赖**：不需要 API Key，不需要网络请求，离线可用
- **零成本**：不需要付费调用 API
- **100% 可控**：输出永远符合预期格式

**缺点**：
- **缺乏理解能力**：对"最近有什么好玩的户外活动吗"这种间接表达无法识别（虽然包含"活动"，但不含"运动"关键词）
- **无法处理自然语言**：对多轮对话、上下文、同义词完全无能为力
- **扩展性差**：新增一个标签/功能需要修改代码、重新打包发布


### 11.3 最终方案：Prompt 强约束 + 三层解析降级

**选择理由**：
1. **实现成本低**：一次 HTTP 请求 + 几行 try-catch 就能完成
2. **满足学习目标**：确实调用了 DeepSeek API，体验了完整的 Prompt 设计 → API 请求 → 响应解析 → 数据库检索链路
3. **健壮性**：三层降级保证不会因模型输出不符合预期而崩溃

### 11.4 API Key 安全策略

### 11.5 SearchService 的路由归一化与容错

**searchMethod 归一化匹配**（`SearchService.kt:92-125`）：

```kotlin
val lowerMethod = searchMethod.lowercase()
return when (lowerMethod) {
    "searchbytag", "searchByTag" -> if (tags.isNotEmpty()) searchByTag(tags[0]) else emptyList()
    "searchbymultipletags", "searchByMultipleTags" -> searchByMultipleTags(tags)
    "searchbychannel", "searchByChannel" -> if (tags.isNotEmpty()) {
        val channel = tags[0].toIntOrNull() ?: 0
        searchByChannel(channel)
    } else emptyList()
    "searchbykeyword", "searchByKeyword" -> if (tags.isNotEmpty()) searchByKeyword(tags[0]) else emptyList()

```

这样即使模型返回了大小写不一致的方法名（如 `SearchByTag`、`SEARCH_BY_TAG`）或错误的方法名（如 `findByTag`），也能正确路由或返回空列表。

### 11.6 缓存策略全景

整个项目的缓存设计分布在四个层次：

| 层级 | 缓存对象 | 实现方式 | 缓存粒度 | 生命周期 |
|-----|---------|---------|---------|---------|
| **Room 数据库缓存** | 预置广告数据 | SQLite + Room ORM | 整条记录 | 应用卸载前永久 |
| **Glide 图片缓存** | 图片 Bitmap / 原图文件 | 内存缓存（LruResourceCache） + 磁盘缓存（DiskLruCache） | 单张图片 | 内存：LRU 淘汰；磁盘：应用卸载或手动清理 |
| **AndroidVideoCache 视频缓存** | 视频 mp4 文件 | 本地 HTTP 代理 + 文件缓存 | 单个视频文件 | 最多 20 个文件 / 512MB，LRU 淘汰 |
| **SearchService 全量检索结果** | 数据库查询结果 | 无显式缓存，依赖 Room 内部查询缓存 | 一次查询结果 | 查询后立即释放，不跨请求保留 |

**视频缓存的工作原理**（`VideoPlaybackManager.kt:17-34`）：
- 用户点击播放按钮时，VideoView 的 setVideoURI 接收的不是原始 URL，而是 `getProxyUrl(context, originalUrl)` 返回的 `http://127.0.0.1:xxxx/originalUrl`
- AndroidVideoCache 库在应用内启动了一个微型 HTTP 服务器，监听 localhost 的某个端口
- VideoView 请求 localhost URL 时，代理服务器判断该 URL 是否已在缓存目录存在：
  - **命中**：直接从本地文件读取字节流，通过 HTTP 响应返回给 VideoView（零网络消耗，秒开）
  - **未命中**：向原始 URL 发起下载请求，同时将字节流一边写入本地缓存文件、一边推送给 VideoView（边下边播）
- 缓存目录下文件数超过 20 个或总大小超过 512MB 时，按 LRU 淘汰最旧的文件

**Glide 图片缓存的隐式优化**（`FeedAdapter.kt:143-148`）：
- Glide 默认配置了 `MemoryCache`（约为应用可用内存的 1/20），同一 URL 的图片第二次出现时直接从内存读取，不需要解码
- `DiskCacheStrategy.AUTOMATIC`（默认值）会缓存原图到磁盘，下次打开应用也不需要重新下载
- 本项目中 132 条数据的图片总数有限（约 1-2MB 每张），基本能全部命中内存缓存，滚动体验流畅

**SearchService 未缓存的权衡**：
- 当前每次调用 SearchService 的方法都会重新查询数据库（`dao.getAllItemsByChannel(0)` + 内存 filter）
- 由于本项目数据规模极小（每频道最多 44 条），查询 + filter 在低端机上也能在 10ms 内完成
- 如果数据量增长到万条级别，需要增加 LRU 缓存层：把 `(method, params) → List<AdItem>` 的映射缓存到内存，5 分钟内相同查询直接返回缓存结果

## 十五、设计模式与架构亮点

### 15.1 设计模式总结

| 设计模式 | 应用位置 | 价值 |
|---------|---------|-----|
| **单例模式** | AppDatabase、UserManager、VideoPlaybackManager、FilterManager | 全局统一管理，避免资源冲突和重复创建 |
| **观察者模式** | FilterManager | 标签状态变化时自动通知所有订阅者刷新 |
| **基类继承** | BaseChannelFragment → Channel_1/2/3 | 三个频道复用分页、刷新、过滤、滚动逻辑 |
| **ListAdapter + DiffUtil** | FeedAdapter、MessageAdapter、UserItemsAdapter | 智能差异更新，避免全列表刷新 |
| **DAO 模式** | AdItemDao、UserDao | 数据访问与业务逻辑解耦 |
| **Repository 模式** | SearchService | 封装多种数据库检索方法供上层调用 |

### 15.2 数据流向设计

```
Room 数据库 (ad_items / users)
      ↓
DAO 接口 (AdItemDao / UserDao)  ← 异步查询（Dispatchers.IO）
      ↓
业务层管理器 (UserManager / FilterManager / VideoPlaybackManager)
      ↓
UI 层 (Activity / Fragment + RecyclerView + Adapter) ← 主线程更新（Dispatchers.Main）
      ↓
用户交互 (点击/滚动/输入) → 回调至业务层 → 更新数据库 → 循环
```

## 十六、核心数据交互链路

### 16.1 点赞状态同步（跨列表/详情页）

```
FeedAdapter 点赞按钮
      ↓
切换状态 + animateButton(scale_bounce)
      ↓
dao.updateLikeAndCount(item.id, newLiked, newCount) （Dispatchers.IO）
      ↓
if (newLiked) UserManager.addLike(item.id) else removeLike(item.id)
      ↓
DetailActivity 返回
      ↓
ActivityResultLauncher 回调 → adapter.updateItemState()
      ↓
DiffUtil 检测差异 → 局部刷新 UI
```

### 16.2 浏览历史记录

```
点击卡片 → DetailActivity.onCreate()
      ↓
UserManager.addBrowseHistory(item.id)
      ↓
history.remove(itemId) + history.add(0, itemId) （去重 + 最新优先）
      ↓
if (history.size > 50) history.removeLast() （保留最近 50 条）
      ↓
GlobalScope.launch(Dispatchers.IO) { userDao.updateUser(user) }
      ↓
UserProfileActivity.onResume() → 从数据库按 ID 列表查询 → 更新三个面板
```

### 16.3 AI 对话检索

```
DialogActivity 用户输入
      ↓
AiClient.sendMessage(userMessage, callback)
      ↓
OkHttp 请求 → DeepSeek API /chat/completions
      ↓
Retrofit + GsonConverter 解析 → DeepSeekResponse → AiResponse
      ↓
SearchService.executeSearch(aiResponse.searchMethod, aiResponse.tags)
      ↓
searchByTag / searchByMultipleTags / searchByChannel / searchByKeyword
      ↓
Dao.getAllItemsByChannel(0/1/2) → 合并 → filter 匹配
      ↓
返回 List<AdItem>
      ↓
UI 更新：AI_WITH_SEARCH 类型消息 + 结果卡片列表
      ↓
点击卡片 → 跳 DetailActivity
```
