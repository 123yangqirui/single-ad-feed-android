## 一、棘手问题与解决方案

### 问题 1：手机状态栏遮挡按钮/内容

**现象**：构建无报错，但运行在真机上，顶部内容被系统状态栏挡住，按钮点击不到，布局整体上移。

**根因**：应用使用 `NoActionBar` 主题但没有处理 System UI Insets（系统窗口占用空间），Android 从 4.4 开始状态栏属于系统区域，应用内容默认会延伸到其下方。

**解决方案**（推荐做法）：

```xml
<!-- 方案 A：在根布局设置 fitsSystemWindows -->
<LinearLayout
    xmlns:android="http://schemas.android.com/apk/res/android"
    android:fitsSystemWindows="true"        <!-- 让内容不延伸到状态栏下方 -->
    android:paddingTop="?attr/actionBarSize" <!-- 额外预留一条 ActionBar 高度 -->
    ... >
```

### 问题 2：RecyclerView 多线程数据不一致导致崩溃

**现象**：随机出现 `java.lang.IndexOutOfBoundsException: Inconsistency detected. Invalid view holder adapter positionViewHolder` 或 `IllegalStateException: Cannot call this method while RecyclerView is computing a layout or scrolling`。

**根因**：子线程（协程 IO 线程）修改了 dataList，同时主线程 RecyclerView 正在布局（measure/layout 阶段），两个线程读到的 item 数量不一致。

**本项目中的触发场景**：
- `FeedAdapter` 使用 `ListAdapter + DiffUtil`，但 footer 状态（`showFooter`）改变了 `getItemCount()` 的返回值，若 footer 状态与数据列表不是同一时刻提交，就会不一致。
- `BaseChannelFragment.loadNextPage()` 中，如果 IO 线程读取数据后，切换到主线程提交之前，`isLoading` 标志被另一线程修改，也会导致重复提交。

**解决方案**：
adapter的数据类表，严格限制好主线程和子线程对于数据列表的修改

---

### 问题 3：动画放在 notifyDataSetChanged() 前面被打断

**现象**：点赞/收藏按钮的点击动画（`scale_bounce`）肉眼看不见，或者执行到一半就结束了。

**根因**：`animateButton(view)` 先触发，紧接着 `notifyDataSetChanged()` 或 `submitList()` 被调用。RecyclerView 一收到 notify 就会重新 measure/layout 所有可见 item，视图被重新绑定，刚刚绑定到该 View 上的动画自然就被销毁了。

**解决方案**（按效果从好到差排序）：
先更新数据和提交 adapter，然后再启动动画

---

### 问题 4：ViewPager2 中的 Fragment 频繁销毁重建

**现象**：切换到频道 3 再切回频道 1，频道 1 又从顶部开始加载，浏览位置丢失。

**根因**：ViewPager2 默认只缓存当前页 + 左右各 1 页（`offscreenPageLimit` 默认值在新版中是 -1，即按 `DEFAULT_OFFSCREEN_PAGES = 1` 处理）。三频道切换时中间的页会被销毁重建。

**本项目的修复**：

```kotlin
viewPager2.offscreenPageLimit = 2  // 缓存当前页 + 左右各 1 页 = 共 3 页常驻
```

### 问题 5：Room 依赖安装迷糊 + 网络问题

**现象**：`build.gradle.kts` 里加了 `implementation("androidx.room:room-runtime:2.6.1")`，但 `@Database`、`@Dao` 注解报红找不到，或者编译时报 "Schema export directory is not set"。

**原因**：
1. **缺少编译器插件**：Room 需要注解处理器（annotation processor），AndroidX 官方推荐用 KSP。没加 `ksp("androidx.room:room-compiler:2.6.1")` 就会找不到 `AppDatabase_Impl` 这个 KSP 生成的类。
2. **国内网络问题**：google()、mavenCentral() 下载慢或超时，KSP 插件下载失败。
3. **schema 输出目录未配置**：`exportSchema = true` 时必须配置 `ksp { arg("room.schemaLocation", "$projectDir/schemas") }`，否则报警。

**完整正确配置**（`app/build.gradle.kts`）：

```kotlin
plugins {
    alias(libs.plugins.android.application)
    alias(libs.plugins.kotlin.compose)
    id("com.google.devtools.ksp")  // 必须！
}

dependencies {
    implementation("androidx.room:room-runtime:2.6.1")
    implementation("androidx.room:room-ktx:2.6.1")    
    ksp("androidx.room:room-compiler:2.6.1")           
}

```

使用国内镜像：在 `settings.gradle.kts` 顶部将 google() 和 mavenCentral() 替换为阿里云镜像（注意：新版本 AGP 推荐使用 `dependencyResolutionManagement`）：

```kotlin
dependencyResolutionManagement {
    repositoriesMode.set(RepositoriesMode.FAIL_ON_PROJECT_REPOS)
    repositories {
        maven("https://maven.aliyun.com/repository/google")
        maven("https://maven.aliyun.com/repository/public")
        google()       // 留一个 fallback
        mavenCentral() // 留一个 fallback
    }
}
```

### 问题 6：Room 不支持 List<String> 等复合类型字段

**现象**：编译报错 `Entities and POJOs must have a usable public constructor` 或 `Cannot figure out how to read this field from a cursor`。

**根因**：Room 只能直接映射基本类型（Int、String、Long 等），`List<String>` 这种复合类型必须手动转换。

**解决方案**：使用 `TypeConverter` + `Gson`（本项目已实现，位于 `dataprocess/ListConverter.kt`）：

```kotlin
// ListConverter.kt --------------------------------------------------
class ListConverter {
    @TypeConverter
    fun fromList(list: List<String>): String = Gson().toJson(list)

    @TypeConverter
    fun toList(json: String): List<String> =
        Gson().fromJson(json, object : TypeToken<List<String>>() {}.type)
}
```

 .build()
```

### 问题 8：主线程操作数据库导致 ANR

**现象**：点击登录按钮白屏 5 秒以上，Logcat 出现 `android.os.NetworkOnMainThreadException` 或类似主线程卡死现象。

**根因**：Room 默认不允许在主线程执行数据库操作（除非调用 `.allowMainThreadQueries()`，但这是毒药）。

**本项目中的表现**：`UserManager.login()` 用了 `runBlocking(Dispatchers.IO) { ... }`，虽然内部切到了 IO 线程，但 `runBlocking` 本身会阻塞调用它的线程（主线程）。登录操作量小可能还能过，但如果查询量一大就会 ANR。

**正确修复方案**：

```kotlin
// 推荐：使用 suspend 函数，让调用方用 lifecycleScope 启动
object UserManager {
    suspend fun login(username: String, password: String): Boolean {
        return withContext(Dispatchers.IO) {
            val user = userDao.getUserByUsername(username)
            user != null && user.password == password
        }
    }
}

// LoginActivity 中：
btnLogin.setOnClickListener {
    lifecycleScope.launch {  // ← 跟随 Activity 生命周期，Activity 销毁就取消
        val success = UserManager.login(username, password)
        if (success) { /* 跳转 */ }
    }
}
```

**额外**：`runBlocking` 只适合在 `main` 函数或单元测试中用，`GlobalScope.launch` 也应避免使用，它脱离了生命周期管理，容易泄漏（本项目 FeedAdapter 中 `GlobalScope.launch(Dispatchers.IO)` 就是一个改进点——Activity/Fragment 销毁了但协程还在跑，可能持有 View 引用导致内存泄漏）。

### 问题 9：不使用全局单例会有多个视频同时播放

**现象**：列表中有 5 个视频卡片，用户依次点击播放 1、播放 2、播放 3……结果 3 个视频同时出声，界面卡顿，音量混杂。

**根因**：每个 VideoView 是独立对象，点击播放按钮时直接调用 `videoView.start()`，没有一个"裁判"来管理播放状态。

**本项目的解决方案**：`VideoPlaybackManager` 全局单例（`util/VideoPlaybackManager.kt`）

```kotlin
object VideoPlaybackManager {
    private var currentPlayingId: String? = null
    private var currentVideoView: VideoView? = null

    fun play(itemId, videoView, playButton, coverView, videoUrl) {
        // 关键：新视频播放前，先暂停旧视频
        if (currentPlayingId != itemId) {
            pauseCurrent()  // 暂停前一个
        }
        // 更新当前播放引用
        currentPlayingId = itemId
        currentVideoView = videoView
        // ... 播放逻辑（设置代理 URL、准备、启动）
    }
}
```

### 问题 10：视频每次播放都从网络重新加载，体验差

**现象**：WiFi 下还好，移动网络或弱网下每个视频都要等 3-5 秒缓冲，用户体验极差。

**解决方案**：引入 **AndroidVideoCache**（`com.danikula:videocache:2.7.1`），用本地 HTTP 代理做缓存：

```kotlin
// VideoPlaybackManager.kt 中
private fun getProxy(context: Context): HttpProxyCacheServer {
    if (proxyCacheServer == null) {
        synchronized(this) {
            if (proxyCacheServer == null) {
                proxyCacheServer = HttpProxyCacheServer.Builder(context)
                    .cacheDirectory(context.cacheDir)   // 缓存目录
                    .maxCacheSize(512 * 1024 * 1024)     // 最多 512 MB
                    .maxCacheFilesCount(20)              // 最多 20 个文件
                    .build()
            }
        }
    }
    return proxyCacheServer!!
}

// 使用时把原始 URL 替换成代理 URL
val proxyUrl = getProxyUrl(context, originalUrl)  // http://127.0.0.1:...
videoView.setVideoURI(Uri.parse(proxyUrl))
```

**工作原理**：在本机启动一个微型 HTTP 服务器，VideoView 向 localhost 请求，代理服务器判断是否有本地缓存，有就直接返回文件，没有就边下载边写入缓存边向客户端推流（流式代理）。

### 问题 11：协程线程切换没整明白导致 ANR 或数据不刷新

**现象**：
- 场景 A：IO 操作写在 `Dispatchers.Main` 中 → 网络/数据库请求阻塞主线程 → ANR
- 场景 B：UI 更新写在 `Dispatchers.IO` 中 → 子线程改 UI → `CalledFromWrongThreadException`
- 场景 C：忘记切换回主线程 → 数据加载完了但界面没变化，用户以为没加载

**正确的协程线程切换模板**（本项目中部分代码已遵循，但有几处 GlobalScope 的问题）：

```kotlin
// 在 Activity/Fragment 中（带生命周期管理）：
lifecycleScope.launch {          // 默认在 Main 线程启动
    // 1. 先跑在主线程：显示 loading
    progressBar.visibility = View.VISIBLE

    // 2. 切 IO 线程执行耗时操作
    val data = withContext(Dispatchers.IO) {
        AppDatabase.instance.adItemDao().getItemsSync(channel, 0, 11)
    }

    // 3. withContext 结束后自动切回 Main，更新 UI
    adapter.submitList(data)
    progressBar.visibility = View.GONE
}

// 在 ViewModel 中（更推荐，跟随 ViewModel 生命周期）：
viewModelScope.launch {
    val data = withContext(Dispatchers.IO) { dao.getItems(...) }
    _uiState.value = data  // LiveData/StateFlow 会自动把更新发到主线程
}
```

## 三、踩坑复盘：架构与决策思考

### 3.1 架构设计层面

#### 复盘 1：最初设计——主界面直接用 RecyclerView+Adapter 复用 3 遍是灾难性的

**初始方案**：
- 主 Activity 一个 RecyclerView + FeedAdapter
- 切换频道 = 清空 adapter 现有数据 + submit 新频道数据
- 详情页用 Fragment 弹出

**遇到的问题**：
- 浏览位置丢失
- 数据加载冲突
- 无法实现"频道切换动画"
- 详情页生命周期复杂


**最终方案**：`MainActivity + ViewPager2 + 3 个 Channel Fragment + 独立的 DetailActivity`，每个 Fragment 有自己的 Adapter、分页状态、加载状态，互相不干扰。

---

#### 复盘 2：后端服务直接调用大模型 API —— 是否需要中间层？

**初始想法**：APP → 自建后端（FastAPI）→ 大模型 API，理由是"保护 API Key"、"做后端聚合"。

**放弃原因**：
- Demo 场景下，多一层网络调用徒增延迟（本地 → 后端 → 大模型，多一跳）
- 部署和维护额外的服务器成本（学生场景下尤其不现实）
- 真正生产环境才需要：后端做鉴权、限流、缓存、审计、API Key 保护
- 本地开发环境：直接从 APP 调用 DeepSeek 更简单

## 四、后续可改进点

### 改进 1：替换 GlobalScope / runBlocking

**问题定位**：`FeedAdapter.kt:446/462`、`UserProfileActivity.kt:221`、`ListConverter.kt`（UserManager）大量使用了 `GlobalScope.launch` 和 `runBlocking`。

**思路**：

```
- `GlobalScope` 没有生命周期管理 → Activity/Fragment 销毁后协程还在跑，可能持有 View 引用导致内存泄漏。
- `runBlocking(Dispatchers.IO)` → 虽然内部切 IO，但 `runBlocking` 本身阻塞调用线程（主线程），登录/注册操作量大时会 ANR。

**改造模板**：

```kotlin
// 1) UserManager.login/register 改为 suspend，由调用方用 lifecycleScope 启动
object UserManager {
    suspend fun login(username: String, password: String): Boolean =
        withContext(Dispatchers.IO) {
            val user = userDao.getUserByUsername(username)
            user != null && user.password == password
        }
}

// 2) LoginActivity 中：
btnLogin.setOnClickListener {
    lifecycleScope.launch {
        val success = UserManager.login(username, password)
        if (success) { /* 跳转 */ }
    }
}

// 3) FeedAdapter 中 GlobalScope.launch → 改为通过回调让 Fragment 的 lifecycleScope 启动
//    Adapter 不自己开协程，把点赞/收藏的 IO 操作通过回调交给上层处理
```

### 改进 2：密码哈希

**问题定位**：`ListConverter.kt:37/51` 直接明文存储和比较 `user.password == password`。

**思路**：使用 BCrypt 或 PBKDF2 做单向哈希，数据库只存 hash，登录时对用户输入同样哈希后比较。

```kotlin
// 依赖（使用 Spring Security 的 BCrypt 或 AndroidX Security 库）
// 在 register 时：
val hashedPassword = BCrypt.hashpw(plainPassword, BCrypt.gensalt(12))
userDao.insert(User(username, hashedPassword, ...))

// 在 login 时：
val hashed = BCrypt.hashpw(inputPassword, storedSalt)  // 或直接用 BCrypt.checkpw
BCrypt.checkpw(inputPassword, storedHash)
```

**注意**：salt 每次注册时随机生成并与 hash 一起存储，避免彩虹表攻击。

### 改进 3：登录状态持久化

**问题定位**：目前没有登录状态管理，关闭 APP 后每次都要重新登录。

**思路**：用 `SharedPreferences` 或 `EncryptedSharedPreferences`（更安全）存一个 token 或 sessionId，启动时检查是否有效。

```kotlin
// 登录成功后写入：
val sp = context.getSharedPreferences("app_prefs", Context.MODE_PRIVATE)
sp.edit().putString("session_token", token).apply()

// Splash/启动页检查：
val token = sp.getString("session_token", null)
if (token != null && isValid(token)) {
    // 直接跳 MainActivity
} else {
    // 跳 LoginActivity
}
```

**额外**：退出登录时清除 token；token 加一个过期时间（如 7 天）避免永久有效。

### 改进 4：View 引用泄漏保护

**问题定位**：`VideoPlaybackManager` 的 `currentVideoView` 是强引用；FeedAdapter 中的 View 可能被长时间持有。

**思路**：

- `VideoPlaybackManager` 使用 `WeakReference<VideoView>` 避免 Activity 销毁后还持有 View 强引用。
- Fragment/Activity `onDestroyView/onDestroy` 时调用 `VideoPlaybackManager.release()` 主动释放。

```kotlin
object VideoPlaybackManager {
    private var currentVideoViewRef: WeakReference<VideoView>? = null

    fun play(...) {
        currentVideoViewRef = WeakReference(videoView)
    }

    fun release() {
        currentVideoViewRef?.get()?.stopPlayback()
        currentVideoViewRef = null
        currentPlayingId = null
    }
}

// Fragment onDestroyView 中调用 release()
```

### 改进 6：图片加载优化

**问题定位**：Glide 使用简单 `placeholder/error`，移动网络下体验一般。

**思路**：

- **缩略图策略**：小图用 `thumbnail(0.2f)` 先加载模糊缩略图。
- **缓存策略**：根据网络类型做不同处理（WiFi 预加载大图，移动网络只加载缩略图）。
- **RecyclerView 滚动时暂停加载**：`RecyclerView.addOnScrollListener`，滚动时 `Glide.with(context).pauseRequests()`，停止时 `resumeRequests()`，避免滚动卡顿。

```kotlin
recyclerView.addOnScrollListener(object : RecyclerView.OnScrollListener() {
    override fun onScrollStateChanged(rv: RecyclerView, newState: Int) {
        if (newState == SCROLL_STATE_SETTLING || newState == SCROLL_STATE_FLING) {
            Glide.with(context).pauseRequests()
        } else {
            Glide.with(context).resumeRequests()
        }
    }
})
```

### 改进 7：性能监测

**思路**：

- **StrictMode**：开发环境开启 `StrictMode.ThreadPolicy.Builder().detectAll().penaltyLog()`，检测主线程读写磁盘/网络。
- **Flipper**：Facebook Flipper 可视化网络、数据库，便于调试。
- **LeakCanary**：检测 Activity/Fragment 泄漏，接入后很容易发现 VideoPlaybackManager 等全局引用泄漏。
- **Perfetto/Systrace**：追踪启动耗时、列表滑动 FPS。

```kotlin
// Application onCreate 中（仅 debug 构建）
if (BuildConfig.DEBUG) {
    StrictMode.setThreadPolicy(
        StrictMode.ThreadPolicy.Builder().detectAll().penaltyLog().build()
    )
    StrictMode.setVmPolicy(
        StrictMode.VmPolicy.Builder().detectLeakedSqlLiteObjects().detectLeakedClosableObjects()
            .penaltyLog().penaltyDeath().build()
    )
}
```

### 改进 8：日志分级 + 统一 Logger

**问题定位**：目前可能零散的 `Log.d/Log.e` 散布在各处，release 版不区分级，难以定位和统一开关。

**思路**：封装一个统一的 Logger 工具类，release 自动关闭 debug/verbose 日志，仅保留 error/warn。

```kotlin
object AppLogger {
    private const val TAG = "MyApp"
    fun d(msg: String) { if (BuildConfig.DEBUG) Log.d(TAG, msg) }
    fun e(msg: String, tr: Throwable? = null) { Log.e(TAG, msg, tr) }
    // ... 其它级别
}

// 使用：AppLogger.d("load data: $size")
